# Stage 4 — Static Site for Human Browsing

## Overview

A MkDocs Material static site generated from the guardrail metafiles in this
repo. Architects, engineers, and other stakeholders can browse guardrails by
category, search by keyword or tag, and click through to source documents —
without needing access to GitHub or understanding the metafile schema.

The site is generated from the same `guardrails/**/*.json` files that already
power Stage 2's `guardrails-index.json`. A new script (`generate_docs.py`)
reads those files and writes Markdown pages. MkDocs Material builds the
static HTML. A GitHub Action runs both steps on every merge to `main` and
deploys to GitHub Pages.

No new dependencies on the intake pipeline. No changes to the schema or
existing metafiles. Stage 4 is purely additive.

---

## What gets built

```
docs-site/
├── mkdocs.yml                        # Full site config (nav regenerated by script)
└── docs/
    ├── index.md                      # Home: overview, stats, recently updated
    ├── tags.md                       # Tag index (rendered by MkDocs Material)
    └── guardrails/
        ├── governance/
        │   ├── index.md              # Category overview: description + table of guardrails
        │   ├── GUARD-GOV-001.md
        │   ├── GUARD-GOV-002.md
        │   └── ...
        ├── security/
        │   ├── index.md
        │   └── ...
        ├── cloud/
        ├── api/
        └── architecture/
```

GitHub Action deploys the built HTML to the `gh-pages` branch.

## Design decisions

**Status filtering**: Include `approved` and `under-review` guardrails. Exclude
`draft` (incomplete) and `deprecated` (superseded). Status is shown as a badge
on each page.

**One page per guardrail**: Each metafile gets one Markdown file at a
predictable path: `docs/guardrails/{category}/{ID}.md`. Filenames match the
guardrail ID (e.g. `GUARD-GOV-001.md`).

**Nav is fully regenerated**: `generate_docs.py` writes the entire `mkdocs.yml`
from a static template merged with the dynamic nav section. Do not hand-edit
`mkdocs.yml` — it will be overwritten.

**Tags via frontmatter**: MkDocs Material's built-in `tags` plugin reads `tags:`
from each page's YAML frontmatter. The `tags` array from the metafile is passed
directly. A `/tags` index page is generated automatically.

**Related guardrails as links**: The `related_guardrails` array is rendered as
a list of Markdown links pointing to the corresponding `.md` files. If a
referenced ID does not exist in the site (e.g. it is still draft), the link
is rendered as plain text rather than a broken link.

**Change log inconsistency handling**: Some metafiles use `changelog` (no
underscore) and `change` instead of `summary` in log entries. `generate_docs.py`
handles both variants when rendering the change history table.

## Module detail — `generate_docs.py`

**Entry point**: `python generate_docs.py` from repo root.

**Inputs**: `guardrails/**/*.json` (excluding `guardrails/schema/`)

**Outputs**: all files under `docs-site/` (the directory is cleared and
rebuilt from scratch on each run)

**Constants** (top of file):

```python
GUARDRAILS_DIR = Path("guardrails")
SCHEMA_DIR = GUARDRAILS_DIR / "schema"
DOCS_SITE_DIR = Path("docs-site")
DOCS_DIR = DOCS_SITE_DIR / "docs"
INCLUDE_STATUSES = {"approved", "under-review"}
```

### Step 1 — Load and filter metafiles

Identical logic to `generate_index.py`. Walk `GUARDRAILS_DIR.rglob("*.json")`,
skip `SCHEMA_DIR`, load JSON, filter to `INCLUDE_STATUSES`. Build a list of
dicts. Also build:

- `by_category: dict[str, list[dict]]` — metafiles grouped by category
- `id_to_path: dict[str, str]` — maps guardrail ID to its relative docs path
  (e.g. `"GUARD-GOV-001"` → `"../governance/GUARD-GOV-001.md"`) for
  cross-linking

### Step 2 — Clear output directory

Delete `docs-site/docs/` entirely and recreate it. This prevents stale pages
from lingering if a guardrail is deprecated (removed from the filtered set)
between runs.

Do **not** delete `docs-site/mkdocs.yml` separately — it is overwritten in
Step 5.

### Step 3 — Write per-guardrail pages

For each metafile, write `docs-site/docs/guardrails/{category}/{ID}.md`.

**YAML frontmatter** (read by MkDocs for search and tags):

```yaml
---
title: "{ID} — {title}"
tags:
  - {tag1}
  - {tag2}
search:
  boost: 1.0
---
```

**Page body** (in order):

1. `# {title}` (H1)

2. Status badge inline with version:
   `**Status:** ✅ Approved` / `🔄 Under Review` / `⚠️ Draft` / `❌ Deprecated`
   and `**Version:** {version}` on the same line, separated by two spaces.

3. Metadata table:

   | Field | Value |
   |---|---|
   | ID | `{id}` |
   | Category | `{category}` (capitalised) |
   | Subcategory | `{subcategory}` or — |
   | Type | `{type}` |
   | Owner | `{owner}` or — |
   | Approved by | `{approved_by}` or — |
   | Approved date | `{approved_date}` or — |
   | Review due | `{review_due}` or — |

4. `## Description` section: the `description` field as a paragraph.

5. `## Source document` section:
   - For `onedrive` source: `[{filename}]({url})` (links to the OneDrive URL)
   - For `external_url` source: `[{page_title}]({url})` (links to the external URL)
   - Show last modified/fetched date in italics below the link.

6. `## Tags` section: each tag as a `#tag` link (MkDocs Material tag syntax) or
   a plain comma-separated list if the tags plugin handles them via frontmatter
   only. Since the tags plugin renders them automatically from frontmatter,
   **omit this section** to avoid duplication. The tags will appear in the
   page header automatically.

6. `## Related guardrails` section (only if `related_guardrails` is non-empty):
   For each ID in the list, look it up in `id_to_path`. If found, render as
   `- [{ID} — {title}]({relative_path})`. If not found (draft/deprecated),
   render as `- {ID}` (plain text, no broken link).

7. `## Change history` table:

   | Version | Date | Author | Summary |
   |---|---|---|---|
   | {version} | {date} | {author} | {summary or change} |

   Handle the `changelog` / `change_log` and `summary` / `change` field
   name variants by checking both keys when reading each entry.

### Step 4 — Write category index pages

For each category in `by_category`, write
`docs-site/docs/guardrails/{category}/index.md`.

**Frontmatter**: `title: "{Category} Guardrails"` (no tags).

**Page body**:

1. `# {Category} Guardrails` (H1, capitalised)

2. Count line: `{n} guardrail(s) — {n_approved} approved, {n_under_review} under review.`

3. A Markdown table listing all guardrails in the category, sorted by ID:

   | ID | Title | Status | Version | Review due |
   |---|---|---|---|---|
   | [GUARD-XXX-001](GUARD-XXX-001.md) | {title} | ✅ Approved | 1.0 | 2026-11-01 |

### Step 5 — Write home page (`docs-site/docs/index.md`)

**Frontmatter**: `title: "Architecture Guardrails Library"`.

**Page body**:

1. `# Architecture Guardrails Library` (H1)

2. Brief intro paragraph (hardcoded — not derived from metafiles):
   "This library contains the approved architecture guardrails for [organisation].
   Guardrails are standards, policies, patterns, and rules that guide how systems
   are built and integrated. Browse by category, search by keyword, or filter by
   tag."

3. Stats block:
   ```
   **{total} guardrails** across {n_categories} categories —
   {n_approved} approved, {n_under_review} under review.
   Last updated: {today's date}.
   ```

4. Category summary table:

   | Category | Guardrails | Approved | Under Review |
   |---|---|---|---|
   | [Governance](guardrails/governance/index.md) | 36 | 30 | 6 |
   | [Security](guardrails/security/index.md) | 8 | 8 | 0 |
   | ... | | | |

5. `## Recently updated` section: the 10 most recently modified guardrails,
   sorted by `approved_date` descending (fall back to `change_log[0].date`
   if `approved_date` is null):

   | ID | Title | Category | Date |
   |---|---|---|---|
   | [GUARD-GOV-037](guardrails/governance/GUARD-GOV-037.md) | ... | Governance | 2026-03-28 |

### Step 6 — Write tags page (`docs-site/docs/tags.md`)

A minimal page that triggers MkDocs Material's tag index rendering:

```markdown
---
title: Tags
---

# Tags

[TAGS]
```

The `[TAGS]` marker is the special token that MkDocs Material's `tags` plugin
replaces with the full tag index at build time.

### Step 7 — Write `docs-site/mkdocs.yml`

Write the entire file from scratch. Structure:

```yaml
# DO NOT EDIT — regenerated by generate_docs.py on every run.

site_name: Architecture Guardrails Library
site_description: Approved architecture guardrails — browse by category, search by keyword.
docs_dir: docs
site_dir: ../site

theme:
  name: material
  palette:
    scheme: default
    primary: indigo
    accent: indigo
  features:
    - navigation.tabs
    - navigation.sections
    - navigation.top
    - search.suggest
    - search.highlight
    - content.tooltips

plugins:
  - search
  - tags:
      tags_file: tags.md

nav:
  - Home: index.md
  - Tags: tags.md
  - Governance:
    - guardrails/governance/index.md
    - GUARD-GOV-001 — {title}: guardrails/governance/GUARD-GOV-001.md
    - ...
  - Security:
    - guardrails/security/index.md
    - ...
  - Cloud:
    - guardrails/cloud/index.md
    - ...
  - API:
    - guardrails/api/index.md
    - ...
  - Architecture:
    - guardrails/architecture/index.md
    - ...
```

The `nav` section is built dynamically. Within each category, guardrails are
sorted by ID. Nav entry titles use the format `{ID} — {title}` (truncated to
60 chars if longer, with `…` appended).

The `site_dir: ../site` setting outputs the built HTML to `site/` at the repo
root (sibling to `docs-site/`), keeping generated HTML out of `docs-site/`.
Add `site/` to `.gitignore`.

## GitHub Actions workflow — `generate-guardrails-docs.yml`

Triggers on push to `main` when guardrail files or the generator script change.
Runs the generator, builds the MkDocs site, and deploys to GitHub Pages.

```yaml
name: Generate and deploy guardrails site

on:
  push:
    branches:
      - main
    paths:
      - "guardrails/**"
      - "generate_docs.py"
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pages: write
      id-token: write

    steps:
      - name: Checkout repo
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: pip install mkdocs-material

      - name: Generate docs from metafiles
        run: python generate_docs.py

      - name: Build MkDocs site
        run: mkdocs build --config-file docs-site/mkdocs.yml

      - name: Commit generated docs to main
        uses: stefanzweifel/git-auto-commit-action@v5
        with:
          commit_message: "chore: regenerate guardrails docs [skip ci]"
          file_pattern: docs-site/docs/** docs-site/mkdocs.yml
          commit_user_name: "github-actions[bot]"
          commit_user_email: "github-actions[bot]@users.noreply.github.com"

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./site
          commit_message: "chore: deploy guardrails site [skip ci]"
```

**Note on sequencing**: The auto-commit step runs first (commits the generated
`.md` sources back to `main` for traceability), then the deploy step publishes
the built HTML to `gh-pages`. Both steps run in the same job so there is no
risk of the deploy running against an empty `site/` directory.

**Why commit the `.md` files to `main`**: Keeps generated Markdown in source
control for auditability — reviewers can diff what changed in the docs when a
guardrail is updated. Mirrors the existing pattern used for `guardrails-index.json`.

## One-time manual setup

Before the workflow can deploy to GitHub Pages:

1. In the repo, go to **Settings → Pages**
2. Set **Source** to **Deploy from a branch**
3. Set **Branch** to `gh-pages`, folder `/` (root)
4. Save

The `gh-pages` branch is created automatically by `peaceiris/actions-gh-pages`
on the first workflow run.

## Local development workflow

```bash
# Install deps (if not already)
uv pip install mkdocs-material

# Generate pages from metafiles
python generate_docs.py

# Serve locally with hot reload
mkdocs serve --config-file docs-site/mkdocs.yml

# Or build static HTML to site/
mkdocs build --config-file docs-site/mkdocs.yml
```

The `mkdocs serve` command watches `docs-site/docs/` for changes and reloads
automatically. Note: since `generate_docs.py` regenerates all files, you must
re-run it manually (then MkDocs will detect the changes) — `mkdocs serve` does
not invoke the generator.

## Known limitations and out of scope

1. **No OneDrive document preview** — the source link points to OneDrive, not an
   embedded preview. Viewers need OneDrive access to open the actual document.
2. **Manual org name** — the intro paragraph on the home page has `[organisation]`
   as a placeholder. This should be updated to the actual organisation name before
   going live.
3. **No status promotion UI** — promoting a guardrail from `draft` to `approved`
   still requires editing the JSON metafile in GitHub directly.
4. **GitHub Pages only** — the plan uses GitHub Pages for deployment. For a
   corporate deployment, swap the `peaceiris/actions-gh-pages` step for a push
   to Azure Static Web Apps (the rest of the pipeline is identical).
5. **Search is client-side only** — MkDocs Material's search index is built at
   build time and loaded as a static JS bundle. This is sufficient for a library
   of this size (~50 guardrails) but would need reconsideration above ~1,000 pages.
